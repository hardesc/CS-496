    Assignment 3: Part 1

    Submit Assignment

    This week there are no deliverable in terms of code (but you are strongly encourage to try some test implementations to see how your database works). Instead you will work on the first part of a 2 part project.

    At the end of next week you will turn in a web service that has a functional, publicly consumable API. What this means is that you will publish a set of URIs clients can use to interact with your web service. Up until now you have handled all the interaction through a web interface that you designed. Now the idea is that you will publish an API and others can write applications which utilize your back-end.

    As a part of the requirement for this, you will need to store data. To do this you will design a schema for a non-relational database, that is a database which does not support the use of joins.

    This weeks portion of the assignment is as follows:

    • Describe the data you will be modeling
      • For this assignment you will need to have at least two types of things with at least two properties each, though something a bit more complex than this would be ideal. These things should be related, for example products and shipments, classes and students etc.
    • Come up with a plan to model the data in a non-relational database.
      • You should discuss how the database stores data, how you will set up the database to store your data and why you chose to set it up the way you did
    • Research one additional non-relational database.
      • Discuss the differences in how data is stored and what impact that would have on the way you choose to model the data.

    The lectures will go into a fair amount of detail on how to use the Google App Engine NDB to model data and the types of things one must consider with this particular database. You can use this as one of your databases you discuss, either as the one you actually implement or as the one your compare it to. The requirement to look at two database is specifically to require you to look at at least one database not covered in class.

    I would expect this to be a 2-3 page PDF document. They types of considerations will vary greatly depending on the constraints of the database you are using and the use case for your application. If you find that you are not coming up with a compelling use case, make one up that is interesting, pick something that will be unreasonably popular and optimize for that use.
